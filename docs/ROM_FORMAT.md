# `.gw` package format and conversion

This application accepts compact `.gw` packages generated by
LCD-Game-Shrinker from a user's own legally obtained MAME Game & Watch ROM and
artwork files. The project does not include ROMs, converted packages, Nintendo
artwork, source artwork, or links to copyrighted downloads.

Madrigal/Libretro `.mgw` files are simulations, not original Game & Watch
program ROM packages. They are intentionally unsupported and are rejected with
an on-device explanation.

## Conversion workflow

1. Obtain the matching Game & Watch ROM set and MAME artwork through lawful
   means. Keep these files outside this repository.
2. Clone or download LCD-Game-Shrinker from its upstream project.
3. Follow that project's dependency setup and place the ROM archive and its
   matching artwork archive in the input locations documented there.
4. Run `shrink_it.py` for the MAME machine identifier. The current enabled
   identifiers are `gnw_ball` and `gnw_flagman`. Leave
   `flag_background_jpeg = False`; the current Playdate renderer accepts the
   resulting RGB565 background but not the optional appended JPEG form yet.
5. Copy the generated `.gw` file to the Playdate data disk at:

   ```text
   /Shared/Emulation/game-and-watch/games/
   ```

The ROM picker creates that directory if it does not exist. Filenames may be
changed, but package identity is determined from the internal machine signature,
CPU type, program size, and program CRC32—not from the displayed filename.

Ball and Flagman are execution-enabled at the current proof-of-concept stage;
Ball has been validated on physical hardware and Flagman has a loader-validated
package pending its device test. The other 23 requested titles are present in
the strict whitelist database but return a safe “not enabled yet” error until
their individual display and input behavior is implemented and tested.

## Binary layout

LCD-Game-Shrinker writes a little-endian payload, normally inside a standard LZ4
frame. An optional JPEG background may follow the LZ4 frame. Uncompressed
payloads beginning with `SM5` are also recognized.

The decoded payload starts with this 108-byte header:

| Offset | Size | Meaning |
| ---: | ---: | --- |
| 0 | 8 | CPU name (`SM5A___\0`, `SM510__\0`, `SM511__\0`, or `SM512__\0`) |
| 8 | 8 | Last eight bytes of the MAME machine identifier |
| 16 | 7 | Hour/minute/second RAM addresses and PM-bit mask |
| 23 | 1 | Reserved; must be zero |
| 24 | 4 | Hardware/rendering flags |
| 28 | 80 | Ten little-endian `(offset, size)` object pairs |

The ten object pairs are, in order:

1. 320×240 RGB565 background (or size zero when the JPEG flag is set)
2. packed LCD-segment pixels
3. 256 32-bit segment pixel offsets
4. 256 16-bit segment X coordinates
5. 256 16-bit segment Y coordinates
6. 256 16-bit segment heights
7. 256 16-bit segment widths
8. optional melody ROM
9. SM5xx program ROM
10. ten 32-bit keyboard-matrix mappings

Objects are four-byte aligned. Segment pixels may use 8, 4, or 2 bits per pixel
according to header flags. Coordinates describe one 320×240 display. This port
does not interpret SVG on the device and rejects multi-screen/unknown packages
before CPU execution.

## Validation and limits

The Playdate loader explicitly decodes little-endian fields and validates the
LZ4 descriptor, every block boundary, decoded-size limit, object range and
ordering, alignment, fixed metadata counts, every segment rectangle and packed
pixel span, keyboard size, machine/CPU whitelist pair, program size, and program
CRC32. User-controlled format errors return to an error screen; they do not use
assertions.

Current safety limits are 2 MiB for the selected file and 400,000 bytes for the
decoded payload. The decoder requires independent LZ4 blocks and a stored
content size, matching LCD-Game-Shrinker's current output.
